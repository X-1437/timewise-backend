
# 第二周需求文档：前端基础搭建

## 1. 需求背景
"鸿溯"时间序列数据分析助手第一周已完成技术方案验证，确定了Python MCP框架 + OpenAI API的技术路线。第二周需要搭建前端基础原型，实现对话界面、CSV文件上传功能，并验证前端与MCP Resources功能的打通，为后续开发奠定前端基础。

**重要原则**：采用"MVP核心 + 预留扩展"策略，先跑通流程，同时为未来迭代预留架构空间。

## 2. 功能需求

### 2.1 前端基础界面（两栏布局，类似ChatGPT）
需要实现两栏布局（左侧侧边栏 + 右侧主聊天区），包含以下核心组件：

| 区域 | 组件 | 功能描述 | 技术实现 | 优先级 | 备注 |
|------|------|----------|----------|--------|------|
| **左侧侧边栏** | **新建会话按钮** | 点击创建新会话 | React按钮 | P0 | 这周实现 |
| | **会话列表** | 展示历史会话，点击切换 | React列表 | P0 | 这周实现（内存存储） |
| **右侧主聊天区** | **对话输入框** | 用户可以输入自然语言文本 | React表单组件 | P0 | 已实现 |
| | **消息展示区** | 展示用户和助手的消息 | React列表组件 | P0 | 已实现 |
| | **文件上传按钮** | 用户可以点击选择CSV文件 | HTML input type="file" | P0 | 已实现 |
| | **上传状态提示** | 显示文件上传进度/状态 | React状态管理 | P0 | 已实现 |

### 2.2 CSV文件上传功能
需要实现CSV文件的上传和临时存储功能：

| 功能点 | 详细描述 | 验收标准 | 优先级 |
|--------|----------|----------|--------|
| **文件选择** | 支持用户从本地选择CSV文件 | 只能选择.csv后缀的文件 | P0 |
| **文件上传** | 将文件上传到服务器临时文件夹 | 文件成功保存到服务器 | P0 |
| **文件验证** | 验证文件格式是否为CSV | 非CSV文件提示错误 | P0 |
| **上传反馈** | 显示上传成功/失败状态 | 用户能看到明确的状态提示 | P0 |

### 2.3 MCP Resources功能集成
需要实现MCP Resources功能，让大模型能够读取用户上传的CSV文件：

| 功能点 | 详细描述 | 验收标准 | 优先级 |
|--------|----------|----------|--------|
| **文件注册** | 将上传的CSV文件注册为MCP Resource | MCP能发现该文件 | P0 |
| **文件读取** | 大模型可以通过MCP读取CSV文件内容 | 能返回文件内容预览 | P0 |
| **规则匹配** | 用规则匹配模拟大模型回复（不买token） | 多种查询都能得到回复 | P0 |

**规则匹配测试用例**（不只是一句！）：

| 用户输入 | 预期回复 | 验证目的 |
|---------|---------|---------|
| "这个CSV里有什么数据" | "我读取到了你的CSV文件，里面有XX行数据，包含XX、XX等字段" | 验证MCP Resources能读取文件 |
| "帮我看看数据概览" | "好的，我来帮你做EDA分析（模拟）：数据时间范围是XXX到XXX，平均值是XXX..." | 验证意图识别流程 |
| "上传成功了吗" | "是的，文件已上传成功，文件名是XXX.csv，大小是XXXKB" | 验证状态反馈 |
| "你好" | "你好！我是鸿溯时间序列数据分析助手，请上传CSV文件开始分析吧" | 验证基础对话 |

### 3.3 规范的API设计（工程化）

#### 3.3.1 API接口规范
| 接口名称 | 方法 | URL | 请求参数 | 响应格式 | 说明 | 状态 |
|---------|------|-----|---------|---------|------|------|
| **上传文件** | POST | `/api/v1/files/upload` | `file: File` | `{ id: string, filename: string, size: number }` | 上传CSV文件 | 第二周实现 |
| **获取文件列表** | GET | `/api/v1/files` | - | `[{ id: string, filename: string, uploadedAt: string }]` | 获取已上传文件 | 第二周实现 |
| **发送消息** | POST | `/api/v1/chat/messages` | `{ content: string, fileId?: string }` | `{ id: string, role: 'user'|'assistant', content: string }` | 发送对话消息 | 第二周实现 |
| **流式消息** | WS | `/api/v1/chat/stream` | - | 流式推送 | WebSocket实时通信 | 第三/四周预留 |
| **获取会话列表** | GET | `/api/v1/sessions` | - | 会话列表 | 历史会话 | 第三周预留 |
| **创建会话** | POST | `/api/v1/sessions` | `{ title: string }` | 会话对象 | 新建会话 | 第三周预留 |

#### 3.3.2 数据结构规范（TypeScript）
```typescript
// 消息类型
interface Message {
  id: string;
  role: 'user' | 'assistant';
  content: string;
  timestamp: Date;
  fileId?: string;
  toolCalls?: ToolCall[];
}

// 文件类型
interface UploadedFile {
  id: string;
  filename: string;
  size: number;
  uploadedAt: Date;
  mimeType: string;
  url?: string;
}

// 会话类型（第三周预留）
interface Session {
  id: string;
  title: string;
  messages: Message[];
  files: UploadedFile[];
  createdAt: Date;
  updatedAt: Date;
}

// 工具调用（预留）
interface ToolCall {
  id: string;
  name: string;
  args: Record&lt;string, any&gt;;
  result?: any;
}
```

#### 3.3.3 React表单组件说明
**React表单组件**是对原生HTML输入框的封装，好处是：
- 复用：多处使用无需重复代码
- 统一：样式、行为一致
- 扩展：方便添加新功能

示例：
```typescript
// 封装前（原生HTML）
&lt;input 
  type="text" 
  placeholder="输入消息..." 
  onChange={(e) =&gt; setValue(e.target.value)}
  onKeyDown={(e) =&gt; e.key === 'Enter' &amp;&amp; handleSend()}
/&gt;

// 封装后（React组件）
&lt;ChatInput 
  placeholder="输入你想分析的内容..." 
  onSend={handleSend} 
  disabled={isLoading}
/&gt;
```
