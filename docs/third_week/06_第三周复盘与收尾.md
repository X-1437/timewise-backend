## 1. 第三周目标回顾（对齐）

第三周主题：接入 MongoDB 与调用 MCP。

第三周可验证的最终目标：完成“真实大模型（DeepSeek/OpenAI 兼容）→ Tool Calls → MCP 工具调用 → MongoDB 落库 → 历史回放”的端到端闭环，并形成可复用的分层架构底座，为第四周继续扩展打基础。

- 工程层：形成可复用的分层结构（API/Service/DAL/LLM/MCP），便于第四周继续扩展

## 3. 计划 vs 实际（偏差与原因）

- 计划侧重点：MongoDB 对接 + 调试大模型调用 + 验证 MCP
- 实际强化点：将“可用”升级为“可证明”，补齐了接口回放与 Mongo 存证的端到端证据链
- 原因：避免出现“前端有回复但无法证明 tool calling 与落库”的验收风险

### 3.1 实际执行路径复盘（按链路拆解）

- 第一步先稳住“数据底座”：让 sessions/messages/files/time_series_data 可创建、可查询、可回放
- 第二步再做“工具链路”：让模型产生 tool_calls（选对工具 + 参数结构化）
- 第三步再做“存证”：接口回放能读到 tool_calls，Mongo 里也能读到 toolCalls（便于复盘审计）

### 3.2 计划偏差点（是什么、为什么、影响）

- 偏差点 1：验证“证据链”比“跑通一条 demo”投入更多
  - 原因：只看“前端有回复”不可审计，容易在第四周扩展时掉链
  - 影响：第三周的验证工作更扎实，第四周的回归成本显著下降
- 偏差点 2：从 OpenAI 直接调用切换到 DeepSeek（OpenAI 兼容）
  - 原因：成本/可用性/接入便利性综合考虑（并且能满足 tool calling）
  - 影响：后续统一以 `.env` 的 base_url/model 作为切换开关，验证口径更稳定

## 4. 关键决策与影响（沉淀）

- 大模型默认通道：DeepSeek（OpenAI 兼容模式），减少切换与不确定性
- 证据标准：以 `tool_calls` 为核心，必须满足“发送接口返回 + 历史回放 + Mongo 存证”
- 命名与工程约束：代码侧避免与第三方包命名冲突（相关说明见 [04_第三周任务完成说明.md](file:///d:/trae_workspace/data/file/third_week/04_%E7%AC%AC%E4%B8%89%E5%91%A8%E4%BB%BB%E5%8A%A1%E5%AE%8C%E6%88%90%E8%AF%B4%E6%98%8E.md)）

### 4.1 决策 1：以 tool_calls 作为“完成”证据核心

- 选择原因：它同时证明了“模型做了结构化决策”与“后端走了可追踪流程”
- 带来的好处：
  - 复盘时可以还原：用户输入 → 工具选择 → 参数抽取 → 工具输出
  - 后续可扩展：第四周引入 prompts、更多工具时，可用同一证据口径回归

### 4.2 决策 2：CSV 行级入库到 time_series_data（时间序列集合）

- 选择原因：后续做统计/预测/回放时，数据可按文件维度聚合与追溯
- 与“只存原文件”的差异：
  - 行级入库：支持数据库侧做快速查询、审计与采样；但会占用存储
  - 只存原文件：实现更简单，但难以复盘“模型到底基于哪份数据”

### 4.3 决策 3：分层架构优先（API/Service/DAL/LLM/MCP）

- 选择原因：第三周主题虽然偏“接入”，但第四周会进入“协议&链路”复杂度上升期
- 影响：第三周的结构性投入，使第四周可以更聚焦在协议、提示、链路，而不是反复重构目录
