### 2.2 架构设计原则（产品经理要理解！）

| 原则 | 说明 | 对产品的好处 | 学习目的 |
|------|------|-------------|---------|
| **单一职责** | 每个模块只干一件事 | 改一个功能不影响其他功能 | 学习如何让代码易维护 |
| **依赖倒置** | 上层依赖抽象接口，不依赖具体实现 | 让架构更灵活 | 学习如何设计灵活的架构 |

| **分层清晰** | 层与层之间职责明确 | 问题定位快，新人易上手 | 学习如何组织代码结构 |

## 3. 数据模型详细设计

### 3.1 数据建模流程（产品经理也要参与！）

```
1. 列出所有实体（Entity）
   ↓
2. 定义实体的属性（Attribute）
   ↓
3. 定义实体之间的关系（Relationship）
   ↓
4. 选择数据库类型（SQL vs NoSQL）
   ↓
5. 设计Schema + 索引
   ↓
6. 评审和优化
```

### 3.2 实体-关系图（ER图）

```
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│   Session    │         │   Message    │         │    File      │
│  (会话)     │         │   (消息)     │         │   (文件)     │
├──────────────┤         ├──────────────┤         ├──────────────┤
│ _id (PK)     │◄────────│ sessionId    │◄────────│ sessionId    │
│ title        │  1:N    │ _id (PK)     │  1:N   │ _id (PK)     │
│ createdAt    │         │ role         │         │ filename     │
│ updatedAt    │         │ content      │         │ size         │
└──────────────┘         │ timestamp    │         │ filePath     │
                         │ fileId (FK)  │         │ uploadedAt   │
                         │ toolCalls    │         └──────────────┘
                         └──────────────┘
```

**关系说明**：
- 1个Session → 多个Message（1:N）
- 1个Session → 多个File（1:N）
- 1个Message → 0或1个File（1:0..1）

```javascript
// 假设CSV格式：date, sales, product
// 2026-05-01, 1000, A
// 2026-05-02, 1200, B
{
  "timestamp": ISODate("2026-05-01T00:00:00Z"),  // 时间列（时间序列集合的timeField）
  "metadata": {
    "fileId": ObjectId("666abc123def456789abc125"),  // 属于哪个文件
    "product": "A"  // 其他维度列（metadata里的字段）
  },

{
  "timestamp": ISODate("2026-05-02T00:00:00Z"),

```

- 所有数据都存 MongoDB
- CSV原文件不存本地，只存文件信息
- CSV解析后的每一行存到 time_series_data 时间序列集合

### 3.4 MongoDB Schema详细设计

#### 集合1：sessions（会话表）

```javascript
{
  _id: ObjectId,           // 会话ID（主键）
  title: String,           // 会话标题，默认"新会话"或第一条消息前20字
  createdAt: Date,         // 创建时间
  updatedAt: Date          // 最后更新时间（有新消息时更新）
}
```

**索引设计：**
```javascript
db.sessions.createIndex({ updatedAt: -1 })  // 按更新时间倒序，快速获取最近会话
```

**对应的Python模型：**
```python
from pydantic import BaseModel
from datetime import datetime
from typing import Optional

class Session(BaseModel):
    id: Optional[str] = None
    title: str
    created_at: datetime
    updated_at: datetime
```

---

#### 集合2：messages（消息表）

```javascript
{
  _id: ObjectId,           // 消息ID（主键）
  sessionId: ObjectId,     // 所属会话ID（外键）
  role: String,            // 角色："user" 或 "assistant"
  content: String,         // 消息内容
  timestamp: Date,         // 时间戳
  fileId: ObjectId|null,   // 关联的文件ID（可选）
  toolCalls: [             // 工具调用记录（可选）
    {
      toolName: String,    // 工具名称：eda_analysis / preprocessing / ...
      toolArgs: Object,    // 工具参数：{ data_name: "xxx", ... }
      toolResult: String,  // 工具返回结果
      timestamp: Date      // 工具调用时间
    }
  ]
}
```

**索引设计：**
```javascript
db.messages.createIndex({ sessionId: 1, timestamp: 1 })  // 快速获取某个会话的消息，按时间排序
```

**对应的Python模型：**
```python
from pydantic import BaseModel
from datetime import datetime
from typing import Optional, List, Dict, Any

class ToolCall(BaseModel):
    tool_name: str
    tool_args: Dict[str, Any]
    tool_result: str
    timestamp: datetime

class Message(BaseModel):
    id: Optional[str] = None
    session_id: str
    role: str  # "user" or "assistant"
    content: str
    timestamp: datetime
    file_id: Optional[str] = None
    tool_calls: Optional[List[ToolCall]] = None
```

---

#### 集合3：files（文件表）
```javascript
{
  _id: ObjectId,           // 文件ID（主键）
  filename: String,        // 原始文件名（如：sales_data.csv）
  size: Number,            // 文件大小（字节）
  rowCount: Number,        // CSV有多少行数据
  columnNames: [String],   // CSV的列名数组
  uploadedAt: Date,        // 上传时间
  sessionId: ObjectId|null // 所属会话ID（可选）
}
```

**索引设计：**
```javascript
db.files.createIndex({ sessionId: 1 })  // 快速获取某个会话的文件
```

**对应的Python模型：**
```python
from pydantic import BaseModel
from datetime import datetime
from typing import Optional, List

class File(BaseModel):
    id: Optional[str] = None
    filename: str
    size: int
    row_count: int
    column_names: List[str]
    uploaded_at: datetime
    session_id: Optional[str] = None
```

---

#### 集合4：time_series_data（时间序列集合，存解析后的CSV数据）
```javascript
// MongoDB时间序列集合
{
  timestamp: Date,         // 时间列（timeField，必需）
  metadata: {              // 元数据（metaField，可选）
    fileId: ObjectId,      // 属于哪个文件
    // 其他维度字段，比如 product, region 等（从CSV列解析）
  },
  // 其他指标字段（从CSV列解析）
  // 比如：sales, revenue, profit 等
}
```

**创建时间序列集合：**
```javascript
db.createCollection("time_series_data", {
  timeseries: {
    timeField: "timestamp",  // 指定时间字段
    metaField: "metadata",   // 指定元数据字段
    granularity: "hours"     // 时间粒度：seconds / minutes / hours / days
  }
})
```

**索引设计：**
```javascript
// 时间序列集合会自动优化，不需要额外索引
db.time_series_data.createIndex({ "metadata.fileId": 1 })  // 按文件ID查询
```

**对应的Python模型：**
```python
from pydantic import BaseModel
from datetime import datetime
from typing import Optional, Dict, Any

class TimeSeriesData(BaseModel):
    id: Optional[str] = None
    timestamp: datetime
    metadata: Dict[str, Any]  // { fileId, ...其他维度 }
    # 其他动态指标字段
```

## 4. API接口设计

### 4.1 接口设计原则

- **RESTful风格**：GET（查）、POST（增）、PUT（改）、DELETE（删）
- **统一响应格式**：所有接口返回格式一致
- **清晰的错误码**：前端知道怎么处理错误

### 4.2 统一响应格式

```json
{
  "success": true,
  "data": { ... },
  "message": "操作成功",
  "error_code": null
}
```

错误响应：
```json
{
  "success": false,
  "data": null,
  "message": "会话不存在",
  "error_code": "SESSION_NOT_FOUND"
}
```

### 4.3 接口列表

#### 会话相关接口

| 方法 | URL | 说明 | 请求体 | 响应 |
|------|-----|------|--------|------|
| POST | `/api/v1/sessions` | 创建会话 | `{ title?: string }` | `{ id, title, created_at, updated_at }` |
| GET | `/api/v1/sessions` | 获取会话列表 | - | `[{ id, title, updated_at }, ...]` |
| GET | `/api/v1/sessions/{id}` | 获取会话详情 | - | `{ id, title, created_at, updated_at, messages: [...] }` |
| PUT | `/api/v1/sessions/{id}` | 更新会话 | `{ title?: string }` | `{ id, title, updated_at }` |
| DELETE | `/api/v1/sessions/{id}` | 删除会话 | - | `{ success: true }` |

#### 消息相关接口

| 方法 | URL | 说明 | 请求体 | 响应 |

## 5. 功能需求

### 5.1 MongoDB数据库接入

| 功能点 | 详细描述 | 验收标准 | 优先级 |
|--------|----------|----------|--------|
| MongoDB环境搭建 | 本地安装MongoDB Community Server | MongoDB服务能正常启动 | P0 |
| 数据库连接 | 后端连接MongoDB | 后端启动时能成功连接数据库 | P0 |
| 数据库与集合创建 | 创建数据库timewise，创建sessions、messages、files集合 | 集合创建成功 | P0 |
| DAL层实现 | 实现数据访问层，定义统一接口 | 上层不需要知道用的是Mongo | P0 |

### 5.2 真实大模型接入

| 功能点 | 详细描述 | 验收标准 | 优先级 |
|--------|----------|----------|--------|
| 环境变量配置 | 配置OPENAI_API_KEY | 能从环境变量读取API Key | P0 |
| LLM服务层实现 | 封装大模型调用逻辑 | 能调用OpenAI API | P0 |
| MCP服务启动 | 后端启动时同时启动MCP服务 | MCP服务正常启动 | P0 |
| 工具列表获取 | 从MCP服务获取可用工具列表 | 能获取到eda_analysis等4个工具 | P0 |
| 意图识别与工具调用 | 大模型识别用户意图并调用MCP工具 | 工具调用成功 | P0 |
| 结果整合 | 将MCP工具结果整合到大模型回复中 | 回复包含分析结果 | P0 |
| 工具调用记录 | 将工具调用过程保存到数据库 | 能查看历史工具调用 | P1 |
| 降级方案 | 无API Key时使用规则匹配 | 规则匹配仍能用 | P1 |

### 5.3 完整流程

```
1. 用户打开前端，创建新会话
   ↓
2. 用户上传CSV文件，文件保存到服务器，元数据存入Mongo
   ↓
3. 用户发送消息："帮我看看这个数据的基本情况"
   ↓
4. 后端接收消息，先保存到Mongo
   ↓
5. 后端调用LLM服务：
   - 传入用户消息
   - 传入MCP工具定义
   ↓
6. OpenAI识别意图，选择eda_analysis工具
   ↓
7. 后端调用MCP的eda_analysis工具
   ↓
8. MCP工具返回分析结果
   ↓
9. 后端将工具调用记录保存到Mongo
   ↓
10. 后端将完整回复返回给前端
    ↓
11. 前端展示结果
```

## 附录：产品经理技术词汇表

| 词汇 | 通俗解释 |
|------|---------|
| **Schema** | 数据库的"表头设计"，规定数据怎么存 |
| **索引** | 数据库的"目录"，让查询更快 |
| **CRUD** | Create（增）、Read（查）、Update（改）、Delete（删） |
| **DAL** | Data Access Layer，专门负责和数据库打交道的一层 |
| **RESTful** | 一种API设计风格，用GET/POST/PUT/DELETE分别表示查增改删 |
| **抽象接口** | 定义"要做什么"，但不管"怎么做"，方便切换不同实现 |
