# 第四周回归与验收清单

本清单用于在每次改动后快速回归，确保第四周闭环能力稳定可用，并为验收证据归档提供统一口径。

## 1. 环境前提

- MongoDB 可连接（与 `fourth_week/code/.env` 配置一致）
- 后端服务启动成功（默认 `http://localhost:8000`）
- 前端可选：使用第二周前端或直接用接口测试均可

## 2. 一键回归路径（最小闭环）

### 2.1 创建会话

- `POST /api/v1/sessions`
- 预期：返回 `data.id`（session_id）

### 2.2 上传 CSV

- `POST /api/v1/files/upload`（multipart/form-data）
  - `file`: CSV 文件
  - `session_id`: 上一步 session_id（可选但建议带上）
- 预期：
  - 返回 `data.id`（file_id）
  - 返回 `data.inserted_rows`（插入行数 > 0）

### 2.3 对话触发工具调用（不依赖固定句式）

接口：`POST /api/v1/chat/messages`

请求体关键字段：

- `content`
- `session_id`
- `fileId`

顺序与预期：

1. 表达“确认文件”的意图
   - 预期：触发 `upload_data`
2. 表达“数据概览/基本情况”的意图
   - 预期：触发 `eda_analysis`
3. 表达“预处理”的意图
   - 预期：触发 `preprocessing`
4. 表达“特征分析”的意图
   - 预期：触发 `feature_analysis`
5. 表达“简单预测/基线预测”的意图
   - 预期：触发 `naive_forecast`
6. 表达“导出报告”的意图
   - 预期：触发 `export_markdown`，并返回导出路径

通用验收要求：

- 每一步返回 `success=true`
- assistant 消息返回 `content`（Markdown）
- assistant 消息包含 `tool_calls`，且至少包含 `tool_name/tool_args/tool_result`

### 2.4 拉取消息记录（追溯）

- `GET /api/v1/sessions/{session_id}/messages`
- 预期：能看到 user/assistant 消息序列，assistant 消息包含 tool_calls

### 2.5 验收证据归档

至少归档：

- 一次完整闭环导出的报告文件（`.md`）
- 本次回归使用的 CSV（可选）
- 接口响应样例（可选）

归档路径：

- `D:\trae_workspace\data\file\fourth_week\evidence\`

## 3. 失败定位建议（最常见三类）

1. 上传成功但 EDA/预测无数据
   - 检查 `time_series_data` 是否写入 `metadata.fileId`
2. 对话返回但没有 tool_calls
   - 检查 `ChatService` 是否创建 assistant 消息时带上 tool_calls
3. MCP 调用失败
   - 检查 MCP server 是否能读到 `.env` 并连接 MongoDB
