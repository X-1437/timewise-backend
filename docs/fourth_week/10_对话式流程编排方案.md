# 对话式流程编排方案（第四周问题收敛 + 第五周实现输入）

## 1. 背景

原 Web 形态下，时间序列数据分析助手的核心流程是固定顺序：

1. 上传数据
2. EDA
3. 预处理
4. 特征分析
5. 朴素预测
6. 导出报告

但在当前对话式产品形态下，用户不会天然遵循上述顺序，而是会直接用自然语言发起需求，例如：

- “帮我清洗一下这个数据”
- “帮我简单分析一下这个数据”
- “分析一下这个数据具有哪些特征”
- “进行一下朴素预测”
- “帮我生成分析报告”

这导致原先依赖“页面步骤顺序”的流程约束失效，出现以下问题：

- 用户可能未做 EDA 就直接预处理
- 用户可能未做预处理就直接做特征分析/预测
- 用户可能在预处理后继续做 EDA，但系统未明确说明分析对象是原始数据还是预处理后数据
- 导出报告可能无法准确反映真实分析路径与数据版本

因此，需要为对话式产品建立一套“流程编排机制”，在不破坏自然对话体验的前提下，保证分析流程正确、结果可解释、报告可追溯。

## 2. 核心问题

### 2.1 问题本质

这不是“用户不按流程操作”的问题，而是：

- 对话产品缺少显式流程引导
- 系统未记录当前分析阶段
- 系统未明确标注当前使用的数据版本
- 存在关键步骤被跳过但无提示的情况

### 2.2 当前风险

- 用户误以为系统已使用预处理数据，实际可能仍在使用原始数据
- 用户在未理解数据质量前直接做预测，结论可信度下降
- 报告无法复盘真实执行路径
- 用户体验上缺少“当前做到哪一步”的确定感

## 3. 方案目标

本方案希望达成以下目标：

- 允许用户自由发起对话，不强制要求按 Web 顺序操作
- 系统能识别当前流程阶段，并给出合理引导
- 系统能明确说明每一步使用的是哪一版数据
- 存在流程歧义时，系统必须提示或确认
- 导出报告能够还原用户真实分析路径

## 4. 设计原则

### 4.1 不强制顺序，但必须可感知

用户可以跳步，但系统不能“无声跳步”。

### 4.2 能提示就不硬拦截，必须确认时再确认

大多数场景使用“提示 + 推荐下一步”；
只有在存在明显歧义时才要求用户确认。

### 4.3 所有分析结果必须带数据版本标签

任何 EDA、特征分析、预测、导出结果都必须说明：

- 使用原始数据
- 使用预处理后数据
- 使用的是哪一次预处理产物

### 4.4 报告按真实路径生成，不按理想路径拼接

报告必须忠实记录用户实际走过的步骤，而不是套用固定模板顺序伪造流程。

## 5. 会话状态机设计

建议在会话维度维护分析上下文 `analysis_context`，至少包含以下字段：

| 字段 | 含义 |
|---|---|
| `session_id` | 当前会话 ID |
| `file_id` | 当前分析文件 ID |
| `raw_uploaded` | 是否已有原始数据 |
| `eda_on_raw_done` | 是否已完成原始数据 EDA |
| `preprocess_done` | 是否已完成预处理 |
| `latest_preprocessed_artifact` | 最新预处理产物路径/ID |
| `eda_on_preprocessed_done` | 是否已完成预处理后 EDA |
| `feature_done` | 是否已做特征分析 |
| `forecast_done` | 是否已做预测 |
| `active_data_version` | 当前默认分析数据版本：`raw` / `preprocessed` |

### 5.1 推荐状态流转

| 状态 | 触发动作 | 下一个状态 |
|---|---|---|
| `raw_uploaded` | EDA | `eda_on_raw_done` |
| `raw_uploaded` | 预处理 | `preprocess_done + active_data_version=preprocessed` |
| `preprocess_done` | EDA（预处理后） | `eda_on_preprocessed_done` |
| `preprocess_done` | 特征分析 | `feature_done` |
| `preprocess_done` | 预测 | `forecast_done` |
| 任意分析完成 | 导出报告 | `report_ready` |

## 6. 流程编排规则

### 6.1 用户先做 EDA

- 正常执行
- 输出中明确标注：
  - `数据版本：原始数据`
- 推荐下一步：
  - “建议继续进行预处理，以提升后续特征分析和预测结果的可靠性”

### 6.2 用户先做预处理

- 正常执行，不拦截
- 系统提示：
  - “已完成预处理，后续分析将默认使用预处理后数据”
  - “建议补充一次 EDA，了解处理前后数据质量变化”

### 6.3 用户未做 EDA 就直接做特征分析

- 不强拦截
- 必须提示：
  - “当前尚未查看数据概览，建议先做 EDA 了解缺失值、异常值和分布情况”
- 若继续执行：
  - 结果中标注当前数据版本
  - 追加风险提示：
    - “该结果未经过完整数据概览确认，请结合 EDA 结果解读”

### 6.4 用户未做预处理就直接做特征分析

- 必须先确认，不能直接执行
- 推荐文案：
  - “当前尚未进行预处理，原始数据可能包含缺失值、异常值或类型问题，这会影响特征分析结果。你可以选择先预处理，或继续基于原始数据分析。”
- 提供选项：
  - `先进行预处理（推荐）`
  - `直接基于原始数据分析`
  - `先查看 EDA`
- 交互规则：
  - 系统先弹出选择式确认，不直接返回特征分析结果
  - 只有当用户明确选择“直接基于原始数据分析”时，才继续执行并输出结果
  - 若用户选择“先进行预处理”或“先查看 EDA”，则跳转执行对应步骤

### 6.5 用户未做 EDA 就直接做预测

- 提示强度高于特征分析
- 推荐文案：
  - “当前尚未查看数据概览，建议先做 EDA 确认缺失值、异常值和时间范围，否则预测结果可能缺乏解释基础。”

### 6.6 用户未做预处理就直接做预测

- 必须先确认，且为强确认
- 推荐文案：
  - “当前尚未进行预处理，原始数据中的缺失值、异常值或类型问题可能直接影响预测结果。建议先完成预处理后再进行预测。”
- 提供选项：
  - `先进行预处理（推荐）`
  - `直接基于原始数据预测`
  - `先查看 EDA`
- 交互规则：
  - 系统先弹出选择式确认，不直接返回预测结果
  - 只有当用户明确选择“直接基于原始数据预测”时，才继续执行并输出结果
  - 若用户选择“先进行预处理”或“先查看 EDA”，则跳转执行对应步骤

### 6.7 用户在预处理后再做 EDA

- 必须解决“分析对象歧义”
- 系统必须先进入确认态，不直接返回 EDA 结果
- 推荐确认文案：
  - “当前同时存在原始数据与预处理后数据。你希望我查看哪一份数据的 EDA？”
- 选择项（按钮式确认）：
  - `查看预处理后 EDA（推荐）`
  - `查看原始数据 EDA`
- 交互规则：
  - 系统先展示按钮式确认，不直接输出 EDA
  - 用户点击某一选项后，才执行对应 EDA 并返回结果

### 6.8 用户在已有原始/预处理两份数据时发起泛化请求

例如：

- “帮我分析一下这个数据”
- “看看这个数据怎么样”

此类请求存在强歧义，系统必须先确认：

- 系统必须先进入确认态，不直接执行分析
- 推荐确认文案：
  - “当前存在原始数据与预处理后数据两种分析对象。你希望我基于哪一版数据继续？”
- 选择项（按钮式确认）：
  - `基于预处理后数据继续（推荐）`
  - `基于原始数据继续`
- 交互规则：
  - 系统先展示按钮式确认，不直接输出分析结果
  - 用户点击选项后，系统按所选数据版本继续执行该意图对应的分析（例如 EDA / 特征 / 预测）

### 6.9 用户导出报告

- 报告中必须记录：
  - 实际执行过的步骤
  - 每一步对应的数据版本
  - 是否跳过 EDA
  - 是否跳过预处理
- 若路径存在明显缺口，例如未做 EDA 且未做预处理直接预测：
  - 报告中需增加“结果解读风险提示”

## 7. 数据版本规则

所有分析结果必须在输出顶部明确显示：

- `数据版本：原始数据`
- `数据版本：预处理后数据（preprocessed_xxx.csv）`
- 若跳过预处理：
  - `数据版本：原始数据（未预处理）`

### 7.1 默认策略

- 若用户未指定数据版本，且会话中已存在预处理产物：
  - 对特征分析/预测：默认优先使用最新预处理产物
  - 对 EDA：若同时存在原始数据与预处理后数据，必须先进入确认态（见 6.7）
- 若不存在预处理产物：
  - 使用原始数据
- 若原始数据与预处理数据都存在，且用户请求语义模糊：
  - 必须追问确认

## 8. 交互设计建议

### 8.1 轻量提示卡片

在聊天区增加系统提示卡片，用于展示：

- 当前阶段：已上传 / 已做 EDA / 已预处理 / 可预测 / 可导出
- 当前数据：原始 / 预处理后
- 推荐下一步操作

### 8.2 歧义动作的按钮式确认

当存在路径歧义或存在“确认后才可继续执行”的规则时，统一使用按钮式确认（与当前“未预处理确认”交互一致）：

- `查看原始数据 EDA`
- `查看预处理后 EDA（推荐）`
- `先进行预处理`
- `直接继续分析`

#### 8.2.1 统一返回结构（建议）

- 后端返回 assistant message 时，除 `content` 外，额外返回 `actions` 数组
- `actions` 的每个元素包含：
  - `type`：固定为 `call_tool`
  - `id`：动作唯一标识（用于前端 key）
  - `label`：按钮文案（用户点击时显示/提交）
  - `tool_name`：要执行的工具名（例如 `eda_analysis` / `preprocessing`）
  - `tool_args`：工具参数（必须包含 `session_id` / `file_id`，以及必要时的 `data_version`）
- 前端在聊天气泡下方渲染按钮；用户点击按钮后，前端携带 `action` 再次请求后端，后端直接执行对应工具并返回正式结果（避免二次意图识别造成漂移）

### 8.3 输入框附近增加上下文提示

例如：

- `当前分析对象：预处理后数据（v1）`
- `当前阶段：已完成预处理，建议查看 EDA / 特征分析 / 预测`

## 9. 后端实现建议

### 9.1 新增流程决策层

在“意图识别 -> MCP 工具调用”之间增加“流程决策器”：

1. 识别用户意图（EDA / 预处理 / 特征 / 预测 / 导出）
2. 读取当前会话上下文
3. 判断是否存在前置缺失或数据版本歧义
4. 决定下一步行为：
   - 直接执行
   - 先确认再执行
   - 先追问确认

### 9.2 记录每次 tool call 的数据版本

建议在 `tool_calls` 记录中补充：

| 字段 | 含义 |
|---|---|
| `tool_name` | 工具名 |
| `tool_result` | 工具输出 |
| `data_version` | `raw` / `preprocessed` |
| `artifact_path` | 若使用预处理产物，则记录路径 |
| `warning_flags` | 是否存在未 EDA / 未预处理等风险 |

### 9.3 导出报告增强

报告不再只拼接工具输出，而是按“步骤 + 数据版本 + 风险提示”组织内容。

## 10. 产品规则清单（P0 / P1 / P2）

### 10.1 P0：必须实现

- 所有分析结果标注数据版本
- 未做预处理直接做特征分析时必须先确认
- 未做预处理直接做预测时必须先确认
- 预处理后再做 EDA 时必须确认分析对象
- 导出报告必须记录真实执行路径

### 10.2 P1：建议实现

- 未做 EDA 就做特征分析时给出提示
- 未做 EDA 就做预测时给出提示
- 系统卡片展示当前阶段和推荐下一步
- 提供原始/预处理后 EDA 对比摘要

### 10.3 P2：体验增强

- 可视化流程条
- 智能补步建议
- 自动生成“推荐下一步”快捷按钮

## 11. 验收标准

### 11.1 功能验收

- 用户不按 Web 固定顺序操作时，系统仍能完成闭环
- 预处理后再做分析时，系统明确说明使用的是预处理后数据
- 未预处理直接做特征分析时，系统先弹出选择确认
- 未预处理直接做预测时，系统先弹出选择确认
- 原始数据和预处理数据并存时，系统能处理歧义
- 报告能还原真实分析路径与数据版本

### 11.2 体验验收

- 用户无需记忆固定步骤顺序
- 用户始终知道“现在分析的是哪份数据”
- 系统提示不冗余，不过度打断
- 关键风险点有明确提醒

## 12. 结论

从 Web 形态切换到对话式形态后，原有的“固定流程顺序”必须升级为“可感知、可追溯、可确认的流程编排机制”。

这套方案的核心不是强制用户照流程走，而是：

- 允许自由对话
- 保证关键节点有提示
- 保证数据版本清晰
- 保证报告可复盘

建议将本方案作为：

- 第四周问题复盘的总结项
- 第五周交互与流程优化的正式输入项
