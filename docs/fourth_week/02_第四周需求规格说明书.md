# 第四周需求规格说明书

## 一、文档目的

本说明书用于定义第四周的需求范围、目标用户、核心能力、功能边界、验收标准与执行前提，作为第四周进入研发执行阶段前的正式需求依据。

## 二、需求背景

目前项目已完成前三周基础工作，形成了以下可复用成果：

- 已完成项目总体排期与全流程支撑文档
- 已完成对话驱动产品形态的 PRD 定义
- 已完成前后端基础能力搭建
- 已完成 MongoDB 持久化底座
- 已完成 LLM + tool_calls + MCP 的基础验证

第四周不是重新搭建系统，而是在既有基础上完成“核心协议与核心链路打通”，形成一套可以演示、可以验收、可以继续迭代的最小闭环版本。

## 三、第四周目标

第四周的核心目标为：

1. 固化最小 MCP 工具集与协议
2. 明确与现有后端能力的复用关系
3. 跑通上传、对话触发、工具调用、结果返回、报告导出的标准流程
4. 以朴素预测作为本周预测能力边界
5. 为后续第五周的预览调整与算法校验提供稳定基础

## 四、目标用户

结合已完成的 PRD，本周聚焦以下用户：

- 核心用户：时间序列数据分析师
- 重要用户：有分析诉求但不熟悉复杂工具链的业务人员
- 当前主要使用者：项目开发者本人，用于验证流程、沉淀方法、训练产品能力

## 五、用户核心诉求

目标用户希望系统具备以下体验：

- 能上传一个 CSV 数据文件并开始分析
- 能通过自然语言快速触发分析步骤
- 不需要手动串联复杂的数据分析流程
- 每一步都能获得清晰、可理解的结果输出
- 最终能导出 Markdown 报告用于留档或复盘

## 六、范围说明

### 6.1 本周纳入范围

- CSV 上传
- 对话触发分析步骤
- EDA 分析
- 数据预处理
- 特征分析
- 朴素预测
- Markdown 导出
- 过程记录与结果归档

### 6.2 本周不纳入范围

- 登录/注册/JWT 鉴权
- 复杂预测模型
- PDF 导出
- Excel 上传
- 参数配置中心
- 权限体系与多人协作

## 七、需求清单

### 7.1 数据上传

#### 需求描述

用户上传 CSV 文件后，系统应完成文件接收、基础识别、数据上下文建立，为后续对话分析提供输入基础。

#### 输入

- CSV 文件

#### 输出

- 上传成功反馈
- 文件标识或数据集标识
- 后续会话可引用的数据对象

#### 验收标准

- 至少支持 CSV
- 上传成功后可进入后续分析链路
- 数据对象可被后续工具调用引用

### 7.2 对话触发分析

#### 需求描述

用户使用自然语言描述分析诉求，系统识别意图并选择对应工具执行。

#### 典型表达

- 帮我看一下这个数据的基本情况
- 先做一下预处理
- 分析一下特征
- 做个简单预测
- 导出报告

#### 系统要求

- 优先使用已配置的大模型进行意图识别
- 基于 tool_calls 触发工具
- 不依赖固定关键词或固定句式，允许用户使用任意自然语言表达相同意图
- 参数不完整时可使用默认值
- 触发结果需具备统一结构，便于展示与追踪

#### 验收标准

- 主要场景下可正确映射到目标工具
- 失败时有明确错误反馈或兜底策略

### 7.3 EDA 分析

#### 需求描述

系统支持对已上传数据执行基础探索性分析。

#### 最小输出要求

- 数据规模
- 时间范围
- 缺失值情况
- 基础统计量
- 简要趋势观察

#### 验收标准

- 输出必须基于真实数据
- 返回结果可直接作为对话消息展示

### 7.4 数据预处理

#### 需求描述

系统支持对上传数据执行基础预处理，至少覆盖缺失值处理、异常值处理或类型规范化中的最小能力集合。

#### 最小能力要求

- 至少提供一种默认预处理方式
- 输出处理摘要
- 处理结果可继续进入后续分析步骤

#### 验收标准

- 用户可通过对话独立触发预处理
- 系统返回处理说明与结果摘要

### 7.5 特征分析

#### 需求描述

系统支持对时间序列数据进行基础特征分析，并返回简明结论。

#### 最小输出要求

- 相关性或统计特征
- 趋势特征
- 可解释的简要结论

#### 验收标准

- 结果为结构化内容
- 可直接以 Markdown 形式输出到对话中

### 7.6 朴素预测

#### 需求描述

本周只实现朴素预测能力，作为预测模块的最小基线。

#### 可接受方案

- 前值延续
- 简单移动平均
- 简单趋势外推

#### 输出要求

- 预测方法说明
- 预测周期
- 预测结果

#### 验收标准

- 至少支持对一个样例数据完成未来若干步预测
- 结果可作为对话消息展示

### 7.7 Markdown 导出

#### 需求描述

系统支持将分析过程与结果导出为 Markdown 文件。

#### 最小报告结构

- 数据集信息
- 执行步骤
- EDA 结果
- 预处理结果
- 特征分析结果
- 朴素预测结果
- 生成时间

#### 验收标准

- 成功输出 `.md` 文件
- 结构完整、内容可读

## 八、MCP 最小工具集要求

第四周建议固定为以下最小工具集：

- `upload_data`
- `eda_analysis`
- `preprocessing`
- `feature_analysis`
- `naive_forecast`
- `export_markdown`

每个工具至少需要定义以下内容：

- 工具名称
- 工具描述
- 输入参数 schema
- 默认值策略
- 输出结构
- 错误处理方式

## 九、数据与状态要求

第四周需要保证以下对象在链路中有明确语义：

- 文件对象
- 会话对象
- 消息对象
- 步骤结果对象
- 导出对象

要求：

- 结果应可追溯到步骤
- 会话上下文应可继承
- 导出内容应可基于当前会话生成

## 十、非功能要求

### 10.1 稳定性

- 主链路优先稳定
- 单点失败不能影响整体文档与范围判断

### 10.2 可追溯性

- 每一步应保留步骤语义
- 结果返回应能标识来源工具

### 10.3 可演示性

- 所有能力需服务于“可演示闭环”
- 优先保证主路径顺畅

### 10.4 可学习性

- 需求结构清晰
- 能帮助复盘“为什么本周做这些、不做那些”

## 十一、验收场景

### 场景一：完整闭环

1. 用户上传 CSV
2. 用户在对话框输入任意自然语言，表达“查看数据概览/基本情况”的意图（示例：“先看下数据整体怎么样”）
3. 用户输入任意自然语言，表达“做预处理”的意图（示例：“把缺失值先处理一下”）
4. 用户输入任意自然语言，表达“做特征分析”的意图（示例：“看看趋势和相关性”）
5. 用户输入任意自然语言，表达“做朴素预测”的意图（示例：“给我一个简单基准预测”）
6. 用户输入任意自然语言，表达“导出 Markdown 报告”的意图（示例：“把这次分析导出成报告”）

预期结果：

- 每一步均返回有效结果
- 最终产出 Markdown 报告

### 场景二：单步预测

1. 用户上传 CSV
2. 用户输入任意自然语言，表达“做朴素预测”的意图（示例：“直接给个基线预测结果”）

预期结果：

- 系统执行朴素预测
- 返回预测结果与方法说明

当以下问题均无异议时，可视为第四周需求确认完成，并进入执行操作手册阶段：
