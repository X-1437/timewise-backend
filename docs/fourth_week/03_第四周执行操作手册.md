## 五、第四周核心协议（MCP 工具集）落地步骤

目标：将“对话意图”稳定映射到“工具调用”，并保证工具参数与结果结构可追溯、可导出。

### 5.1 最小工具集（第四周）

为避免“文档工具集”与“代码现状”不一致，第四周按两层口径管理：

- 当前基线（第三周代码现状）：保证能跑通 LLM -> tool_calls -> MCP
- 第四周目标（需求确认口径）：在现状基础上补齐上传/导出与朴素预测

#### 当前基线工具（third_week 基线）

当前 `third_week/code` 的 MCP server 已提供工具：

- `eda_analysis`
- `preprocessing`
- `feature_analysis`
- `forecast`

并且工具入参目前以 `data_name` 为主。

#### 第四周目标工具（需求确认口径）

第四周目标工具集为：

- `upload_data`
- `eda_analysis`
- `preprocessing`
- `feature_analysis`
- `naive_forecast`
- `export_markdown`

其中 `upload_data` 与 `export_markdown` 是为完成闭环新增的工具语义；`naive_forecast` 用于锁定“朴素预测”边界，避免误用复杂模型。

说明：

- 允许用户使用任意自然语言表达意图，LLM 负责意图识别并选择工具
- 工具数量保持最小化，优先保证稳定与可回归

### 5.2 工具参数（建议口径）

为保证“上传 -> 会话 -> 分析 -> 导出”的闭环，工具参数建议统一携带以下最小上下文：

- `session_id`：当前会话
- `file_id`：当前数据文件

各工具建议入参（示例）：

- `eda_analysis`: `{ "session_id": "...", "file_id": "..." }`
- `preprocessing`: `{ "session_id": "...", "file_id": "...", "method": "missing_fill" }`
- `feature_analysis`: `{ "session_id": "...", "file_id": "..." }`
- `naive_forecast`: `{ "session_id": "...", "file_id": "...", "periods": 30 }`
- `export_markdown`: `{ "session_id": "...", "file_id": "...", "scope": "standard_flow" }`

默认值策略：

- `method` 未提供时默认 `missing_fill`
- `periods` 未提供时默认 7 或 30（以需求确认口径为准）

对齐要求：

- 若当前代码工具仍使用 `data_name`，第四周需要在工具层或调用层完成一次参数口径收敛：统一改为 `file_id`（与上传结果一致），并可选携带 `session_id` 用于会话上下文。

### 5.3 工具结果结构（建议口径）

所有工具返回建议满足以下要求：

- 返回 Markdown 文本，可直接展示为对话消息
- 返回文本需包含步骤标题与关键结论
- 返回文本应能被导出模块直接拼装进入报告

### 5.4 与现有后端能力复用（执行口径）

第四周的“真实能力来源”优先级：

1. 优先复用 `timewise-backend` 的已有能力（EDA/预处理/特征/预测/报告等）
2. 若复用成本过高，允许在第四周阶段代码内补充轻量实现，但需在 `evidence/` 中记录原因与取舍

复用映射的结论与接口清单需要归档到 `fourth_week/`（作为验收依据）。

## 六、端到端联调步骤（可回归的最小路径）

目标：形成可重复执行的验证路径，用于每次改动后的快速回归。

### 6.1 创建会话

接口：`POST /api/v1/sessions`

预期：

- 返回 `session_id`

响应外层结构（现状）：

- `success`: boolean
- `data`: 业务数据
- `message`: string
- `error_code`: string | null

### 6.2 上传 CSV 文件

接口：`POST /api/v1/files/upload`（multipart/form-data）

参数：

- `file`：CSV 文件
- `session_id`（可选）：绑定会话

预期：

- 返回 `file_id` 与插入行数等信息

说明：

- 当前上传接口会把 CSV 行写入 MongoDB 的 `time_series_data`（按 `file_id` 关联）
- CSV 时间列会被自动识别（优先匹配 timestamp/datetime/time/date 等字段名，否则使用第 1 列）

### 6.3 发送对话消息触发分析

接口：`POST /api/v1/chat/messages`

请求体（示例结构）：

- `content`: 任意自然语言表达意图
- `session_id`: 上一步会话 id
- `fileId`: 上一步 file id（注意字段名为 `fileId`，不是 `file_id`）

执行顺序（意图级）：

1. 表达“我要看数据概览”的意图 -> 应触发 `eda_analysis`
2. 表达“我要做预处理”的意图 -> 应触发 `preprocessing`
3. 表达“我要做特征分析”的意图 -> 应触发 `feature_analysis`
4. 表达“我要做朴素预测”的意图 -> 应触发 `naive_forecast`
5. 表达“我要导出报告”的意图 -> 应触发 `export_markdown`

预期：

- 每次调用返回一条 assistant 消息内容（Markdown）
- 每次 assistant 消息包含 `tool_calls` 记录（工具名、参数、结果）

### 6.4 拉取消息记录（用于验收追溯）

接口：`GET /api/v1/sessions/{session_id}/messages`

预期：

- 能看到 user/assistant 消息序列
- assistant 消息中包含 `tool_calls`

### 6.5 导出报告（验收关键）

第四周导出要求：

- 至少导出 Markdown 文件
- 文件内容包含：数据集信息、步骤列表、各步骤结果块、生成时间

导出产物存放：

- 统一归档到 `fourth_week/evidence/`
