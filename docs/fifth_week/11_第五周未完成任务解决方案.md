## 3. `W5-08` 导出离线包与报告结构优化

### 3.1 任务背景梳理

- 第五周已确认采用方案 A，即预览与导出统一使用 Markdown，图表统一落 PNG。
- 用户已经连续多轮对下载报告提出结构、可读性、重点突出、CSV 直贴、图前结论等反馈。
- 当前技术链路已具备 ZIP 离线包与报告结构二次优化能力，但仍需最终确认“是否完全满足以后所有导出报告”的口径。

### 3.2 现存问题分析

- 当前问题不再是“能不能导出”，而是“导出体验是否彻底收口”。
- 已做过多轮修正，但最后一版还缺你的最终验收确认。
- 如果不先确认这一步，后续 UAT 和发布文档会建立在不稳定口径上。

### 3.3 分步执行步骤

1. 固化最新报告口径：
   - 顶部摘要信息
   - 原始数据前 5 行样例
   - 预处理后数据样例
   - 每张图片前的看图结论
   - 重点数值加粗
2. 使用标准主流程重新导出一份最新 ZIP。
3. 按“预览一致性、图片本地可见、重点信息可快速扫读、报告结构不累”逐项验收。
4. 将最终通过口径同步写入 UAT 文档与发布前检查清单。
5. 若仍有最后一轮细节问题，只允许做小修，不再大改结构。

### 3.4 所需资源支持

- 最新前后端代码
- `sample_timeseries.csv`
- 你提供的下载报告体验反馈
- 当前 ZIP 导出接口与报告生成逻辑

### 3.5 风险预判及应对措施

- 风险：报告结构继续频繁调整，影响 UAT 与发布收尾。
  - 应对：本轮只做确认与必要小修，不再做大范围重构。
- 风险：不同步骤输出风格不一致。
  - 应对：统一“结论 -> 图片与分析 -> 明细/样例”的报告模板。

### 3.6 验收节点设置

- 节点 1：重新导出的 ZIP 打开后图片全部可见。
- 节点 2：报告顶部可直接看出时间粒度与数据样例。
- 节点 3：每张图前均有基于数据本身的事实性结论。
- 节点 4：你确认“以后所有导出报告都按此口径”。

## 4. `W5-10` 非分析输入统一引导策略代码收口

### 4.1 任务背景梳理

- 第五周文档已经明确将“非分析输入处理”升级为产品规则，而不是继续补示例词。
- 当前代码虽已加入 `direct_reply`，但仍是 `greeting / weather / unsupported_algorithm` 三类特例式处理。
- 这与 SRS 中“分析意图白名单 + 统一引导回复 + 默认不回落 EDA”的正式口径仍有差距。

### 4.2 现存问题分析

- 识别逻辑仍偏“少数 case 命中”，不是“所有非分析输入统一识别”。
- 服务端最终兜底尚未彻底升级为统一引导模板。
- 当前能力会导致边界场景遗漏，存在再次误触发分析链路的风险。

### 4.3 分步执行步骤

1. 梳理当前所有分析意图白名单：
   - 上传/文件操作
   - EDA
   - 预处理
   - 特征提取
   - 朴素预测
   - 导出报告
   - 一键全流程
2. 调整意图识别逻辑：
   - 命中白名单才进入分析工具
   - 未命中白名单统一返回 `direct_reply`
3. 重构 `direct_reply` 内容：
   - 统一主模板
   - 可附加一句“实时信息不可获取”或“当前无该算法工具”
4. 服务端增加最终兜底：
   - 模型未返回工具
   - 工具名无效
   - 输入不属于分析意图
   均统一引导，不默认走 EDA。
5. 补充回归测试：
   - 寒暄
   - 闲聊
   - 天气
   - 身份类问题
   - 未支持算法
   - 完全无关提示词

### 4.4 所需资源支持

- `02_第五周需求规格说明书_SRS.md`
- `03_第五周项目执行操作手册.md`
- 当前 `llm/client.py`、`chat_service.py`
- 历史异常输入样例

### 4.5 风险预判及应对措施

- 风险：白名单过窄，误伤真实分析请求。
  - 应对：以现有单步功能和一键全流程为白名单基础，先保证主能力全部覆盖。
- 风险：白名单过宽，导致无关输入漏进分析链路。
  - 应对：服务端最终兜底保守处理，宁可引导，不误分析。

### 4.6 验收节点设置

- 节点 1：所有非分析输入统一返回同一结构的引导回复。
- 节点 2：任意非分析输入都不再默认回落到 EDA。
- 节点 3：所有分析类请求仍能正常命中对应工具。
- 节点 4：补充 1 份回归结果记录，纳入 UAT。

## 7. `W5-13` 性能实测留痕

### 7.1 任务背景梳理

- SRS 已明确性能目标：
  - 首次响应 ≤ 800ms
  - 复杂任务端到端 ≤ 15s
  - 图表渲染 ≤ 2s
- 当前尚无正式的性能测量记录。

### 7.2 现存问题分析

- 缺少统一计时口径。
- 即使功能正确，也无法证明满足发布门禁。

### 7.3 分步执行步骤

1. 确定计时口径：
   - 首次响应：提交消息到首个有效回复
   - 端到端：提交消息到最终结果完整展示
   - 图表渲染：结果返回到图表可见
2. 选定场景样本：
   - EDA
   - 特征提取
   - 一键全流程
3. 每类场景至少测 3 次，取平均值与最大值。
4. 将结果写入正式记录，附测试环境说明。
5. 若超标，则记录为发布风险或优化项。

### 7.4 所需资源支持

- 浏览器页面
- 后端日志/时间戳
- 手工计时或脚本计时方式

### 7.5 风险预判及应对措施

- 风险：人工计时存在误差。
  - 应对：同一口径重复 3 次，记录平均值与区间。
- 风险：本地环境波动导致结论不稳定。
  - 应对：注明“当前为本地联调环境实测，不作为线上 SLA”。

### 7.6 验收节点设置

- 节点 1：完成 3 类核心场景性能记录。
- 节点 2：形成“满足/部分满足/不满足”结论。
- 节点 3：同步到 UAT 与发布前检查清单。
