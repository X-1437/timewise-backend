## 2. 背景与问题

第五周聚焦“交付前的体验收敛 + 算法可信 + 上线闭环”三件事：

- 右侧预览区承载内容逐步变多（图表、表格、Markdown 报告、导出入口等），需要统一信息架构与交互规则，否则会出现可读性差、定位困难、导出入口分散的问题
- 算法输出需要完成全链路校验，避免 EDA / 预处理 / 建模 / 预测任一环节的偏差导致最终结论失真
- 项目需要走完发布管理流程，把“从需求到上线”的闭环能力固化为可复用的交付模板与产物

## 3. 本周目标

### 3.1 产品目标

- 预览区：在一个稳定的承载框架下同时支持图表、表格、Markdown 报告的浏览与导出，默认路径清晰、内容可定位、错误可解释
- 算法：完成全链路校验与结论输出（发现问题则定位+修复或给出明确限制说明）
- 发布：按“发布前/中/后”流程产出检查清单、回滚预案、发布记录与发布总结

### 3.2 指标目标（验收口径）

性能（暂定指标，按质量体系执行）
- 首次响应：≤ 800ms
- 复杂任务端到端：≤ 15s
- 图表渲染：≤ 2s

质量门禁
- UAT 验收：通过率 100%，并完成签字确认
- 缺陷门禁：发布前修复全部 P0–P1；P2–P3 需具备明确计划

## 4. 范围定义

### 4.1 In Scope（本周必须做）

预览区调整与体验优化
- 预览区承载类型：图表 + 表格 + Markdown 报告
- 统一预览结构：标题/摘要/主体/导出/状态提示
- 可读性：层级、间距、字体、折叠/展开、空/错/加载状态
- 性能：渲染优化策略与关键链路验证（按指标目标）

算法校验（全链路）
- EDA：基础统计、缺失/异常检测、趋势/季节性/平稳性等关键输出一致性检查
- 预处理：缺失值处理、异常处理、缩放/差分等关键步骤输出校验
- 建模/预测：训练集/测试集切分正确性、预测结果合理性、误差指标（如 MAPE/SMAPE/RMSE 等）计算口径一致性
- 输出一致性：预览区展示结果与导出报告/导出文件内容一致

发布闭环
- 发布前检查清单（含回滚预案、监控与告警准备）
- 发布记录（版本、变更、时间点、负责人、结果）
- 发布总结（问题复盘、改进项、后续计划）

### 4.2 Out of Scope（本周不做或降级）

- 新增全新算法能力（仅校验与修复现有链路）
- 大规模 UI 重构（只围绕预览区与导出路径做必要调整）
- 跨团队的外部集成（若需要，作为风险与后续计划记录）

## 5. 核心用户与使用场景

用户角色
- 产品/分析用户：希望快速理解时间序列数据、查看图表与表格、阅读结论报告、导出可复用文档
- 开发/算法用户：希望快速定位算法步骤的输出是否异常，并能复现实验与结论

典型场景
- 用户完成一次分析任务后，在右侧预览区查看“关键图表 + 关键表格 + 报告结论”，并一键导出 Markdown 报告
- 发现预测结果异常时，按步骤回看 EDA 与预处理输出，定位问题来源并复现
- 上线前，按发布清单逐项勾检，形成发布记录与总结，完成全流程闭环

## 6. 功能需求

### 6.1 右侧预览区：信息架构与交互

FR-PRV-001 预览区分层结构
- 顶部：任务名称 / 时间 / 数据集信息（若有）
- 中部：内容区域（支持图表、表格、Markdown 报告三类模块）
- 底部或右上角：导出入口（导出 Markdown / 复制 / 下载等）

FR-PRV-002 内容模块组织规则
- 预览区内至少支持“按模块”组织展示（每个模块独立标题与状态）
- 模块默认排序规则清晰且可复用（例如：概要 → 图表 → 表格 → 报告）

FR-PRV-003 状态体系
- Loading：展示占位骨架或加载提示
- Empty：明确告知当前步骤未生成可预览内容，并给出下一步建议
- Error：给出可定位的信息（错误摘要 + 关键上下文），避免只有“失败”字样

FR-PRV-004 可读性优化
- 支持长内容的折叠/展开或分页策略（避免一次性撑爆容器）
- Markdown 报告支持基本排版（标题/列表/代码块/表格）

FR-PRV-005 内容一致性
- 预览内容与导出内容一致：同一任务、同一版本、同一口径

FR-PRV-006 图表/表格的导出形态（已确认）
- 采用方案 A：导出 Markdown 内引用“静态图片（PNG，chart snapshot）+ 表格（markdown table）”
- 若图片较多：在 Markdown 中按模块分组，并使用分栏排版策略提升可读性

FR-PRV-007 图片产出与可下载（已确认）
- 图片生成：后端生成 PNG 并落盘到 `evidence/` 目录
- 图片可访问：提供图片下载接口（`/api/v1/.../images/{image_id}/download`），Markdown 通过 `![](...)` 引用该 URL
- 可追溯：图片在 MongoDB `artifacts` 集合记录元数据（type/path/sessionId/fileId/createdAt），并记录中文语义名（例如 `displayName`）用于报告呈现与检索

### 6.2 导出与报告

FR-EXP-001 导出 Markdown 报告
- 生成可读的 Markdown 文本，可直接下载
- 允许选择导出范围：
  - 仅标准流程输出
  - 标准流程输出 + 对话结论摘要

FR-EXP-002 导出入口一致
- 导出入口在预览区位置固定、可发现
- 导出过程中有进度/状态反馈，导出失败可重试并可定位原因

FR-EXP-003 导出离线可读（推荐默认）
- 导出报告默认以离线包形式交付（ZIP）：
  - `report_*.md`
  - `images/*.png`
  - `data/preprocessed_*.csv`（若存在预处理产物）
- 离线包解压后，本地打开 `report_*.md` 必须可直接看到图片（使用相对路径引用）
- 保留纯 Markdown 导出能力（不含图片文件），用于在线环境或二次集成

### 6.3 算法校验（全链路）

FR-ALG-001 校验清单机制
- 为 EDA、预处理、建模/预测分别定义“必检项清单”
- 每项校验可输出：输入、关键参数、输出摘要、判定结果（Pass/Fail/Warning）

FR-ALG-002 复现与对照
- 允许选择对照数据集或固定样例数据（若已有），保证可复现
- 误差指标口径一致（例如 MAPE/SMAPE 的分母定义、缺失值处理策略）

FR-ALG-003 问题定位与结论
- 若发现异常，必须输出定位结论：
  - 发生在哪个环节（EDA/预处理/建模/预测）
  - 对最终业务结论影响程度（高/中/低）
  - 修复动作或限制说明

### 6.4 发布闭环（全流程）

FR-REL-001 发布前检查清单
- 代码、测试、依赖、配置、监控、回滚预案等逐项勾检

FR-REL-002 发布记录
- 记录版本、变更范围、执行人、时间点、关键结果、风险与处理

FR-REL-003 发布总结
- 复盘：做得好的、出现的问题、原因分析、改进项
- 形成可复用模板，便于后续迭代直接复用

### 6.5 非分析输入兜底与对话边界

FR-DLG-001 非分析输入统一识别范围
- 系统不能依赖若干固定示例词做兜底，而应识别“所有与数据分析无直接关系的输入”
- 判断原则为：
  - 不属于 EDA、预处理、特征提取、预测、报告导出、数据文件操作等明确分析意图
  - 不要求调用当前系统已支持的数据分析工具链路
- 识别方式应采用“分析意图白名单 + 服务端最终兜底”：
  - 命中分析意图白名单时，才允许进入分析工具链路
  - 未命中分析意图白名单时，统一视为非分析输入

FR-DLG-002 非分析输入响应目标
- 当用户输入与数据分析无关的话语时，系统应统一返回“身份说明 + 能力边界 + 下一步引导”
- 响应不能误触发 EDA、预处理、特征分析、预测、导出等分析工具
- 响应应保持简洁、自然、可继续对话，不得直接报错或沉默

FR-DLG-003 统一响应口径
- 非分析输入不再按多个话术分散处理，而采用统一主模板：
  - 先说明“我是时间序列数据分析助手”
  - 再说明“当前主要支持的数据分析能力范围”
  - 最后给出“下一步你可以怎么做”的引导
- 若用户输入中包含明显的外部实时信息或未知算法诉求，可在统一模板中附加一句简短说明，但整体仍保持同一回复结构

FR-DLG-004 服务端最终兜底
- 即使模型判断错误或未返回有效工具，服务端仍必须对非分析输入进行最终兜底
- 对于无法识别且不属于分析意图白名单的输入，默认返回“统一引导回复”，而非默认执行 EDA
- 只有在明确命中分析意图时，系统才允许进入分析工具链路

### 6.6 标准流程一键运行（主流程自动化）

FR-FLW-001 触发方式（MVP）
- 推荐默认：用户在对话框输入“帮我运行全流程”（或等价表达），系统自动执行标准主流程
- “一键全流程”必须属于分析意图白名单（与 EDA/预处理/特征/预测/导出同级），不得被兜底策略拦截
- 后续可扩展前端按钮入口，但不改变核心流程与输出

FR-FLW-002 标准主流程定义（固定顺序）
- 数据上传确认（绑定当前会话的文件）
- EDA（原始数据）
- 预处理（固定默认方法）
- 特征提取（预处理后数据）
- 朴素预测（相邻两行，预处理后数据）
- 朴素预测（相同时间点，预处理后数据）
- 朴素预测（按天累加，预处理后数据）
- 导出报告（导出范围：标准流程输出）

FR-FLW-003 默认参数策略（MVP 固定）
- 预处理默认方法：异常值处理（outlier_remove）
- 预测默认步数：7
- 目标列：自动选择（与单步运行口径一致）
- 数据版本策略：
  - EDA：原始数据
  - 特征/预测：优先使用预处理后数据；若不存在预处理产物，则需明确提示并回退到原始数据或中止（按执行手册固定）

FR-FLW-004 过程可见性与可追溯
- 全流程执行期间，必须可让用户感知进度（例如按步骤输出“已完成/进行中/失败”）
- 每一步的输出需与单步运行一致（同一工具、同一口径），并可被导出报告复用

FR-FLW-005 失败处理与继续执行
- 若任一步失败：
  - 必须输出失败步骤、失败原因摘要与下一步建议（重试/改参数/先补前置步骤）
  - 后续步骤默认不继续执行（避免产生误导性结果）

FR-FLW-006 全流程导出交付
- 全流程结束后必须自动触发导出，并返回下载链接
- 导出内容与预览一致，并满足 FR-EXP-003 “离线可读”

## 7. 非功能需求

NFR-001 性能
- 满足 3.2 的指标目标，并提供验证方式与记录

NFR-002 可用性与可维护性
- 预览区展示规则在代码层面可复用（避免为每种内容类型写一次性逻辑）
- 算法校验过程可重复执行，校验结果可被归档

NFR-003 安全与合规
- 导出内容不包含敏感信息（若存在脱敏要求，则在执行手册中给出规则）

NFR-004 可解释性与可控性
- 非分析输入的分类与响应规则需可解释、可扩展，便于后续增加更多兜底场景
- 系统默认策略应偏保守：宁可先澄清或给出能力边界，也不能误触发分析结果

## 8. 验收标准（UAT）

预览与导出
- 预览区可稳定展示图表、表格、Markdown 报告三类内容
- 导出 Markdown 可读、可下载、内容与预览一致
- 导出范围可选（标准/含对话结论摘要）

出图清单（最低门槛）
- EDA：可视化至少 1 张
- 预处理：噪音识别至少 1 张；异常值处理至少 1 张
- 特征提取：趋势至少 1 张；时域（周期性/相关性/季节性）至少各 1 张；频域（周期性/傅里叶/小波）至少各 1 张
- 朴素预测：按用户选择的方法（相邻两行/相同时间点/按天累加）至少输出 1 张对应图

性能
- 首次响应、端到端、图表渲染满足指标目标，并有可复查记录

算法
- 全链路校验清单执行完成，有结论文档可追溯
- 若存在缺陷，满足缺陷门禁要求

发布
- 发布前检查、回滚预案、发布记录、发布总结齐全并归档

兜底与对话边界
- 只要输入不属于分析意图白名单，系统统一返回引导性回复，不触发分析工具
- 引导性回复结构统一为“身份说明 -> 能力范围 -> 下一步建议”
- 不再将多个非分析场景拆成不同产品路径，避免规则零散、边界遗漏

一键全流程
- 输入“帮我运行全流程”可自动完成标准主流程，并按步骤输出结果与图表
- 全流程结束后自动导出离线包（ZIP），解压后打开 Markdown 必须可直接看到图片

## 10. 风险与对策

- 风险：2 个工作日内同时覆盖“体验+性能+全链路算法校验+发布闭环”，任务密度较高
  - 对策：优先冻结“预览区目标形态 + 导出口径”，算法校验用清单化快速定位，高风险项先做
- 风险：图表/表格导出形态未定导致实现反复
  - 对策：在你确认导出形态后再进入执行手册阶段落地实现
- 风险：非分析输入识别范围过窄，导致系统把闲聊或无关问题误判为分析请求
  - 对策：将“固定词命中”升级为“输入分类 + 服务端保守兜底”，并纳入独立 UAT 用例池
